# TCP
 네트워크상에 있는 두 기기간에 데이터를 신뢰성 있게 전달하기 위해 만들어진 프로토콜

 ## 특징
  ### 흐름제어
  - 송신측과 수신측의 데이터 처리 속도 차이를 해결하기 위한 기법
  - 송신측의 속도가 빠를 경우 문제가 생기며 수신측에서 제한된 저장 용량을 초과한 이후에 도착하는 데이터는 손실 가능성이 있음.
  - 매번 전송한 패킷에 대해 확인 응답을 받아야만 그 다음 패킷을 전송하는 방법이 있음
  - 수신측에서 설정한 윈도우 크기만큼 송신 측에서 확인 응답없이 패킷을 전송할 수 있게하여 데이터 흐름을 동적으로 조절

  ### 혼잡제어
  - 데이터의 양이 라우터가 처리할 수 있는 양을 초과하면 초과된 데이터는 라우터가 처리하지 못함.
  - 송신 측에서 라우터가 처리하지 못한 데이터를 손실 데이터로 간주하고 계속 재전송하여 혼잡하게하는데 이때 송신 측의 전송속도를 적절히 조절하여 예방 가능
  
  ### 오류제어
  - TCP 통신 중에 오류가 발생하면 해당 데이터를 재전송
  - 송신 측이 ACK을 받지 못하거나, 중복된 ACK를 받는 경우
  - Timeout이 발생하면 이전 데이터를 재전송


  ### 3-way-handshake
  - TCP 통신을 이용하여 데이터를 전송하기 위해 네트워크 연결을 설정하는 과정
  #### 실행순서
  1. SYN 요청
  - SYN을 보내 목적지에 접속을 요청
  2. SYN + ACK
  - 목적지에서 SYN이 들어온 것을 확인하고 SYN+ACK으로 요청을 허가한다고 보낸다.
  3. ACK
  - SYN + ACK을 확인한 출발지에서는 목적지에 ACK를 보내고 연결이 성립된다.
  
  이같이 신뢰성을 위해 3번의 핸드쉐이킹을 거쳐 연결을 맺는다.

  ### 4-way-handshake
  - TCP의 연결된 세션을 종료하기 위해 수행되는 절차
  #### 실행순서
  1. FIN
   - 연결을 종료하겠다는 FIN 플래그를 전송
  2. ACK
   - 목적지에서는 ACK를 보내 알겠다고 전해주면서 종료시킬 준비를 시작한다.
  3. FIN
   - 종료 준비가 다되면 FIN 플래그를 출발지에 다시 전해준다.
  4. ACK
   - 출발지가 해지 준비가 다되면 ACK를 목적지로 다시 보낸다.
# UDP
- 비연결형, 신뢰성이 없는 전송 프로토콜
## 특징
- UDP는 흐름제어, 오류제어와 같이 손상된 것을 재전송하지 않음
- TCP보다 간단하고 빠름
- 흐름제어를 하지 않기 때문에 전송 속도를 최대한 빠르게 함


# HTTP
웹에서 이루어지는 모든 데이터 교환의 기초이며 리소스들을 주고 받을 수 있도록 해주는 프로토콜

  ## HTTP status code
  ### 100 ~ 199 정보 응답
  #### 100 Continue
  진행 중임을 의미하는 응답코드

  #### 101 Switching Protocols
  클라이언트에 의해 보낸 업그레이드 요청 헤더에 대한 응답으로 보내짐

  #### 102 Processing
  서버가 요청을 수신하였으며 이를 처리하고 있지만, 아직 제대로 된 응답을 알려줄 수 없음을 알려줌

  ### 200 ~ 299 성공적인 응답
  #### 200 OK
  요청이 성공적으로 수행

  #### 201 Created
  요청이 성공적이었으며 그 결과로 새로운 리소스가 생성되었음

  #### 202 Accepted
  요청을 수신하였지만 그에 응하여 행동할 수 없는 상태.

  #### 204 No Content
  요청이 성공했지만 클라이언트가 현재 페이지에서 다른 곳으로 이동할 필요가 없음을 나타냄

  ### 300 ~ 399 리디렉션 메시지
  #### 302 Found
  요청한 리소스의 URI가 일시적으로 변경되었음을 의미
   
  ### 400 ~ 499 클라이언트 오류 응답
  #### 400 Bad Request
  잘못된 문법 서버에 요청하여 이해 할 수 없음을 의미

  #### 401 Unauthorized
  요청된 리소스에 대한 유효한 인증 자격이 부족하여 클라이언트 요청이 완료되지 않음을 나타냄

  #### 403 Forbidden
  클라이언트는 콘텐츠에 접근할 권리를 가지고 있지 않음
  401과 다른 점은 서버가 클라이언트가 누구인지 알고 있음

  #### 404 Not Found
  서버는 요청받은 리소스를 찾을 수 없음

  #### 405 Method Not Allowed
  요청한 메소드는 서버에서 알고 있지만 제거 되거나 사용하지 않아서 알려주는 응답

  #### 406 Not Acceptable
  클라이언트와 통신 후 클라이언트에서 정해진 규격에 따른 어떠한 콘텐츠도 찾이 못했을 때
  역직렬화 과정시 무슨 문제가 있는 경우 많이 발생(Getter를 안쓴다거나)

  ### 500 ~ 599 서버 오류 응답
  #### 500 Interneal Server Error
  웹 사이트 서버에 문제가 있음을 의미하지만 서버는 정확한 문제에 대해 더 구체적으로 설명할 수 없음

  #### 502 Bad Gateway
  서버가 게이트웨이로부터 잘못된 응답을 수신, 유효하지 않은 경우

  ## HTTP method
  ### GET
  - 리소스 조회 메소드
  - GET 메소드는 캐싱이 가능하기 때문에 GET을 사용하는 것이 유리

  ### POST
  - 전달한 데이터를 처리/생성 요청하는 메소드
  - 메시지 바디를 통해 서버로 요청 데이터를 전달하면 서버는 요청 데이터를 처리하여 업데이트
  
  ### PUT
  - 리소르르 대체하는 메서드
  - 요청 메세지에 리소스가 있으면 덮어쓰고, 없으면 새로 생성

  ### PATCH
  - 리소스 일부 부분을 변경하는 메소드

  ### DELETE
  - 리소스 제거하는 메소드

  ### HEAD
  - GET과 동일하나 응답 BODY가 없음
  - 오직 있는지 확인할 때만 사용

  ### TRACE
  - 일종의 검사용
  - 서버에 도달 했을 때 최종 패킷의 요청 패킷 내용을 응답 받을 수 있음
  - 변조 유무 확인 가능

  ### OPTION
  - 예비 요청에 사용되는 HTTP 메소드
  - 예비 요청이란 본 요청을 하기 전에 안전한지 미리 검사
  - CORS 정책을 검사하기 위한 요청

  ## HTTP 1.1, 2.0, 3.0
  ### HTTP 1.1
  - 연결당 하나의 요청과 응답을 처리하기 때문에 동시 전송 문제와 다수의 리소스를 처리하기에 속도와 성능 이슈가 존재
  - RTT 증가(양방향 지연)
    - 헤더가 크다
    - 헤더에 많은 메타 정보
    - 중복된 헤더 값을 매 요청시 보냄
  ### HTTP 2.0
  - HTTP 1.1이 느려서 버전을 개선한 것
  - 한 커넥션에 여러개의 메시지를 동시에 주고 받을 수 있음
  - 페이지 로딩 속도 향상
  ### HTTP 3.0
  - TCP가 아닌 UDP를 사용
  - QUIC를 사용(Quick UDP Internet Connection)
    - 연결 레이턴시 감소(요청하고 서버가 처리해 다시 응답해주는 사이클)
    - 패킷 손실 시간 단축
    - 클라이언트의 IP가 바뀌어도 연결 유지
