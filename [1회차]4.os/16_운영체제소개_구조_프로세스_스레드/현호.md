# - 운영체제 구조
##     - 커널과 시스템 호출
- 커널과 쉘이 의사소통하는 수단. 커널은 쉘이 요청할 수 있도록 특정 형태의 통신 인터페이스를 제공하는데, 그것을 시스템콜이라고 한다. 시스템콜은 보통 함수 형태를 갖는다. 시스템콜은 프로그램이 운영체제의 커널에게 서비스를 요청하는 프로그래밍 방식이다.
# - 프로세스
##     - 프로세스 개념
- 운영체제로 부터 자원을 할당받은 작업의 단위
- 모든 프로그램은 운영체제가 실행되기 위한 메모리 공간을 할당해 줘야 실행될 수 있음. 그래서 프로그램을 실행하는 순산 파일은 컴퓨터 메모리에 올라가게 되고, 운영체제로부터 시스템 자원을 할당받아 프로그램 코드를 실행시켜 우리가 서비스를 이용할 수 있게 되는 것.  
##     - 프로세스 상태
- 프로세스의 상태는 프로세스가 실행되는 동안 변경되는 고유 상태를 의미한다. 프로세스가 생성되어 실행하기 까지 프로세스는 여러가지 상태를 갖게 되고, 상태의 변화에 따라 프로세스가 동작되는 것이다. 프로세스는 일반적으로 다음과 같은 5가지 상태를 가진다.
- 생성 : 프로세스가 생성되고 아직 준비가 되지 않은 상태
- 준비 : 프로세스가 실행을 위한 기다리는 상태. CPU를 할당받을 수 있는 상태이며, 언제든지 실행될 준비가 되어있다.
- 실행 : 프로세스가 CPU를 할당받아 실행되는 상태
- 대기 : 프로세스가 IO등의 이벤트가 발생하여 대기하는 상태. CPU를 할당받지 못하며, 이벤트가 마무리 되어 다시 준비 상태로 전환될 때까지 대기한다.
- 종료 : 프로세스가 실행을 완료하고 종료된 상태. 더 이상 실행될 수 없으며, 메모리에서 제거된다.
##     - 프로세스 문맥 교환(컨텍스트 스위칭)
- 컨텍스트 스위칭은 CPU가 한 프로세스에서 다른 프로세스로 전환할 때 발생하는 일련의 과정을 말한다. CPU 한 번에 하나의 프로세스만 실행할 수 있으므로, 여러 개의 프로세스를 번갈아가며 실행하여 CPU활용률을 높이기 위해 컨텍스틑 스위칭이 필요한 것이다.
##     - 프로세스 제어 블록
- OS에서 프로세스를 관리하기 위해 해당 프로세스의 상태 정보를 담고 있는 자료구조를 말한다. 컨텍스트 스위칭이 일어날 때 기존 프로세스의 상태를 어딘가에 저장해 둬야 다음에 프로세스가 돌아왔을 때 같은 작업을 빠르게 이어서 할 수 있을 것이다. 이렇게 상태 정보를 저장해놓기 위해 프로세스 제어 블록은 프로세스 스케줄링을 위해 프로세스에 관한 모든 정보를 저장하는 임시 저장소이다.
- 프로세스 카운터 블록은 프로세스가 메모리에 올라갈 때, 프로세스와 독립된 공간에 함께 올라간다. 프로세스가 종료되면 함께 삭제된다.  
# - 스레드
##     - 스레드 개념
- 프로세스가 할당받은 자원을 이용하는 실행 흐름의 단위  
##     - 멀티스레드의 구조
하나의 프로세스에 두 개 이상의 쓰레드가 존재하는 구조. 더 이상 CPU에서 작업을 처리하는 주체가 프로세스가 아닌 스레스가 된다. 각 스레드는 프로세스로 부터 작업을 위임 받아 CPU를 통해 작업을 처리.  
##     - 멀티스레드의 장단점
장점 :
- 같은 프로세스에 속한 스레드끼리는 컨텍스트 스위칭 비용이 적다. 따라서 CPU가 멀티 태스킹으로 작업을 처리할 때 멀티 프로세스 방식으로 처리할 때에 비해 효율이 좋다.  
- 같은 프로세스에 속한 스레드끼리는 메모리를 공유한다. 따라서 메모리 접근이나 공간 활용적인 측면에서 효율성이 높아진다.(단, 스택, 스택 포인터, 프로그램 카운터 등의 영역은 스레드마다 독립적으로 가지게 됨.)  
단점 :
- 안정성 문제 : 멀티 프로세스 모델에서는 각 프로세스가 독립적으로 동작하므로 하나에 문제가 생겨도 다른 프로세스까지 죽지는 않는다. 그러나 멀티 스레드에서는 하나의 스레드가 영향을 받으면 다른 스레드까지 문제가 생길 수 있다.
